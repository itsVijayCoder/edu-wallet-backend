How the deploy pipeline works

How a deploy starts

Every change reaches production through main on github.com/susanoox/edu-wallet-backend. A poller on the host checks that branch every 5 minutes, so a merge is typically live within 5–7 minutes.

The poller records its runs as auto. A deploy kicked off by hand on the server shows manual instead.

The pipeline

Each attempt runs these phases in order. Everything up to swap is safe — the current version keeps serving traffic until the swap actually happens.

  1. 1
    prechecks

    Validates the compose config, disk space, the port, and the DB migration state. Nothing is touched if this fails.

  2. 2
    build

    Builds the Docker image. Unit tests run inside the build, so a failing test can never reach production.

  3. 3
    backup

    Takes a pre-migration Postgres dump. The last 7 backups are kept.

  4. 4
    migrate

    Applies schema migrations as a one-shot job. It gates the swap — no successful migration, no swap.

  5. 5
    swap

    Replaces the running containers. Expect a few seconds of API downtime at this step.

  6. 6
    postchecks

    Health check, then confirms the running binary reports the exact deployed commit SHA, runs an auth smoke probe, and watches for 30 seconds of stability.

When something goes wrong

  • Failure before the swap — the running version is never touched; the attempt is recorded as Failed.
  • Failure during postchecks — the pipeline automatically reverts to the last known-good image (Rolled back), usually healthy again within about 2 minutes.
  • Two failures in a row of the same commit put deploys in a Held state until a new commit lands — this stops a bad commit from flapping.
  • The database schema is never auto-reverted. Migrations are expected to stay backward-compatible; the pre-migration backups exist for manual recovery.

Reading this page

This page is served by the running API itself, so a failed deploy stays visible — the previous version keeps serving this page. Here is what each state means:

Success the deploy completed and this build is live
Failed stopped before the swap; the old version kept running
Rolled back swapped, failed postchecks, reverted to the last good image
Running the attempt whose commit this server is serving right now
In sync with main the running build equals origin/main HEAD
Differs from main a deploy may be in flight, or local commits are running
Deploys held paused after repeated failures until a new commit lands
Awaiting first poll origin/main has not been recorded yet
